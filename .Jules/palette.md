## 2024-05-24 - Status Indicators Missing Meaningful Context for Screen Readers
**Learning:** Relying purely on visual styling like opacity (e.g., dimming unsupported platform icons) or colors to denote the active/inactive state of an element creates severe accessibility issues. Screen reader users receive no context regarding the difference between a supported and unsupported item when the `alt` text or text content is identical. Furthermore, if a decorative icon next to a text label shares the same `alt` text as the label, it creates redundant announcements (e.g., "Windows, Windows").
**Action:** When implementing status indicators, ensure that dynamic properties update screen-reader-visible text. For icons, ensure the `alt` text actively conveys the current state (e.g., "Supported on Windows" vs. "Not supported on Windows"). For decorative icons placed next to a text label, set `alt=""` and add `aria-hidden="true"` to prevent redundant reading. Also, utilize `sr-only` wrappers for active/inactive text statuses within Badges.
## 2026-04-02 - Dynamic Filtering and Loading Requires Explicit ARIA Live Regions
**Learning:** In a client-side application where search results, pagination, and loading states update asynchronously (e.g., when adjusting filters in a sidebar), screen readers do not automatically announce these DOM changes. A visually obvious "Loading packages..." or "Showing page 2" text update is completely invisible to screen reader users unless explicitly marked as a live region, which can leave them unaware of the active state of their filtering actions.
**Action:** When implementing dynamically updating status or result counts, always wrap or attach `aria-live="polite"` (and `aria-atomic="true"` if necessary) to the container so that screen readers correctly announce the new text content when it changes. Similarly, dynamically rendered error messages should use `role="alert"` (or `aria-live="assertive"`) to ensure immediate announcement.
## 2024-04-09 - Infinite Scroll Lacks Visual Feedback
**Learning:** The infinite scroll mechanism loaded new items silently in the background, offering no visual cue that a fetch was in progress when users reached the bottom of the page. This lack of feedback can lead to confusion, causing users to wonder if theres an issue or if theyve reached the end of the list.
**Action:** Always provide a visual indicator (like a Spinner) and wrap it in an `aria-live="polite"` region when dynamically loading content via infinite scroll. This ensures sighted users see the loading state and screen readers announce it.
## 2024-10-24 - Screen Readers Require Explicit "Opens in New Tab" Context
**Learning:** Providing a visual indicator like `target="_blank"` or an `↗` icon is insufficient for screen readers to recognize that a link navigates away from the current page. Without explicit text context, screen reader users may unintentionally lose their place in the application or become disoriented by unexpected tab behavior.
**Action:** Always append ` (opens in a new tab)` directly to the `aria-label` or include it within a visually hidden `<span className="sr-only">` block for external links.
